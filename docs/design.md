# AgentOS 设计文档

## 1. 产品定位

AgentOS 是一个面向单机环境的 Agent Operating System。

它希望把传统 Agent 从“临时对话助手”升级为“持续运行的软件系统”，让 Agent 不仅能回答问题，还能：

- 理解当前工作区；
- 按能力选择模型；
- 在可控边界内执行任务；
- 跨会话保留记忆；
- 从执行中学习；
- 把高质量经验沉淀为技能。

一句话概括：

- Codex 提供执行纪律；
- Hermes Agent 提供持续进化；
- AgentOS 把两者收敛成一个本地 Agent 操作系统内核。

## 2. 要解决的问题

当前很多 Agent 系统通常落入两种极端：

- 偏聊天：能回答，但不擅长执行、不可观测、上下文容易丢；
- 偏自动化：能执行，但缺乏长期记忆、用户模型、跨任务连续性。

AgentOS 要解决的是：

- 如何让对话、执行、记忆、学习不再是四个孤立子系统；
- 如何让 Agent 的状态可见、可解释、可继承；
- 如何让同一个 Agent 在本地机器上逐渐积累“操作经验”。

## 3. 设计目标

### 核心目标

- 让 Agent 在本地执行时具备明确边界与可见性。
- 让用户偏好、项目规则、历史经验能跨会话保留。
- 让执行结果反向塑造之后的规划与执行质量。
- 让系统先在单机上跑通，再自然升级到多机/多 Agent。
- 让前端和 API 看到的是系统内部状态，而不仅是最终一句回答。

### 次级目标

- 技能可以沉淀为工作区资产；
- 持久层尽量简单，优先 SQLite；
- 策略与权限逻辑尽量放在 runtime，而不是仅靠 prompt；
- 即使模型不可用，也能优雅降级。

### 当前非目标

- 多租户 SaaS 控制面；
- 完整分布式调度；
- 强隔离容器/VM 级安全边界；
- 大规模技能市场；
- 以编辑器协议作为首个主入口。

## 4. 设计原则

### 4.1 本地优先的状态所有权

系统中的关键状态应该都能在本地落盘并被检查：

- tasks；
- sessions；
- memories；
- skills；
- executions；
- learning reports；
- strategy timeline。

这样更利于可信、调试与迁移。

### 4.2 显式的信任边界

Agent 不应该靠“默认全能权限”工作。AgentOS 中的执行能力必须通过显式规则控制：

- 允许哪些程序执行；
- 允许在哪些目录工作；
- 是否可写；
- 执行超时多久；
- 如何取消。

### 4.3 记忆必须结构化

记忆不能只是一个越滚越长的 prompt。AgentOS 要区分：

- 当前会话窗口；
- 会话摘要；
- typed memory；
- 长期策略簇。

### 4.4 学习必须可证伪

如果系统只会积累“经验”，却不知道哪些经验无效，它会越来越迷信自己。

因此 AgentOS 的学习不是只存 lesson，而是要记录：

- 哪个策略被采用；
- 在哪次执行中采用；
- 执行结果如何；
- 是否应降权或 prune。

### 4.5 工作区本身就是输入

仓库中的 `AGENTS.md`、`README.md`、`SOUL.md`、`USER.md`、`MEMORY.md` 都是 Agent 的控制输入，而不是无关文档。

### 4.6 能力要分层，不要混成一个大函数

系统能力需要分成：

- 交互层；
- 控制层；
- 编排层；
- 执行层；
- 存储与反馈层。

这样后续替换前端、接入远程 worker、增加 ACP 才不会推倒重来。

## 5. 用户体验目标

AgentOS 应该满足四种用户直觉。

### 5.1 “它能理解我的仓库”

系统应自动加载项目背景、执行规则和用户偏好，而不是每次重新解释。

### 5.2 “它做的事要能看见”

系统执行任务时，用户应能看到：

- 执行了什么命令；
- 为什么这么做；
- 成功还是失败；
- 输出和审计信息是什么。

### 5.3 “它记得我关心什么”

系统应记住：

- 用户习惯；
- 项目约束；
- 已做决策；
- 常见成功路径。

### 5.4 “它应该越用越顺手”

重复成功的行为应逐渐演化为技能，重复失败的策略应逐渐失去影响力。

## 6. 核心能力设计

### 6.1 任务运行时

任务是 AgentOS 的执行货币，包含：

- 标题与描述；
- 优先级；
- sandbox profile；
- command；
- working_dir；
- strategy_sources；
- execution history。

### 6.2 会话运行时

会话不是消息列表，而是持续协作状态，包含：

- 消息；
- 会话标题；
- 工作目录；
- 压缩摘要；
- pinned decisions；
- 可搜索历史。

### 6.3 记忆运行时

记忆是跨会话的知识单元，支持不同 scope、embedding 和 tags。

### 6.4 技能运行时

技能可以来源于：

- builtin skill；
- 工作区自动发现；
- 从 learning report 晋升而来。

### 6.5 模型路由

不同请求有不同能力需求。AgentOS 通过 capability、default、local-first 等维度进行模型路由，而不是把所有事情交给一个模型。

### 6.6 学习闭环

任务执行完成后，系统会自动产出学习报告、记忆条目、技能候选和策略评估事件。

## 7. 当前约束

### 7.1 SQLite 优先

优点是简单、可移植、低运维；缺点是并发和分布式扩展能力有限。

### 7.2 单机执行语义

当前假设任务由本地子进程执行，而不是远程 worker 集群。

### 7.3 需要启发式降级路径

模型不可用时，AgentOS 仍需依赖 memory/session/skill 检索与启发式规划保持可用。

### 7.4 Prompt 不应承载全部平台规则

执行边界、模型路由、沙箱策略、晋升逻辑尽量由 runtime 明确表达，而不是藏在 prompt 模板里。

## 8. 主要风险

### 8.1 过度学习

如果少量样本就形成高权重策略簇，系统可能被噪声误导。

缓解方式：

- success_rate；
- source_success_rate；
- recency decay；
- suppression；
- prune。

### 8.2 弱隔离被误认为强隔离

当前 sandbox 是应用层策略，不是 OS 级隔离。

缓解方式：

- 文档明确边界；
- 严控 allow-list；
- 后续增加 Docker/SSH/远程 worker 执行器。

### 8.3 状态膨胀

会话、记忆、技能、学习报告增长过快时，系统会难以维护。

缓解方式：

- 保持 typed object；
- 保持显式晋升；
- 区分短期/长期/策略级状态。

## 9. 路线图建议

### Phase 1：单机内核稳定化

- 打磨任务、会话、记忆、技能四大对象；
- 打磨 Dashboard 可观测性；
- 稳定 SQLite schema 与执行路径。

### Phase 2：学习质量增强

- 提升记忆召回排序；
- 优化策略聚类；
- 优化 skill candidate 模板；
- 可视化策略验证/证伪过程。

### Phase 3：接入面扩展

- ACP / IDE 集成；
- Docker / SSH / Remote Worker；
- 更强的 context watch；
- 多 Agent 委派。

### Phase 4：分布式 AgentOS

- 多节点任务调度；
- worker capability registry；
- shared memory policy；
- cluster-level policy。

## 10. 成功判据

当 AgentOS 达到以下状态时，可以认为设计方向正确：

- 用户能看见系统计划了什么、执行了什么、学习了什么；
- 重复任务能逐渐更快或更稳；
- 修改上下文文件即可改变 Agent 行为；
- 模型故障时系统仍有可用退路；
- Skill 不只是生成文件，而是真正可复用的操作资产。
